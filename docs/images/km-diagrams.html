图2：一次对话的完整生命周期

从用户发消息到收到回复，中间经过了7个步骤

1

鉴权 + 加载记忆

JWT 验证身份 → 并行查询三层记忆（L1会话 + L2日志 + L3画像）

~10ms
2

注入灵魂

SOUL.md 精华提取 + Player Card 动态构建 → 拼成 system prompt

~5ms
3

正则门控判断

"提醒我""天气""战绩"等关键词 → 带 Tools 调 LLM
"你好""谢谢"等闲聊 → 不带 Tools，走快通道

<1ms
4

LLM + Function Calling 循环

LLM 自主决定调哪个 Tool → 执行 → 结果回传 → LLM 组织回复
最多 3 轮循环，每层都有超时保护

闲聊 ~2s · 功能 ~5-7s
5

返回响应

reply + chips + calendarEvent + meta → 前端渲染

6

异步更新记忆

L1 追加到会话 · L2 写入日志（不阻塞用户）

异步，不计入响应时间
7

异步画像提取

正则信号快筛（<1ms）→ 命中直接写画像
没命中 → LLM 画像提取（~2s）→ 合并存储

异步，不计入响应时间

图3：灵魂系统三层结构

静态人格 + 身份卡片 + 动态档案 = 一个可信的 AI 人格

👻

SOUL.md — 静态人格

性格（活泼/傲娇/护短）、说话方式（称呼主人、emoji 适度、200字以内）、能力边界、情感规则

启动时加载缓存，extractSoulEssence() 只提取4个关键段落注入 prompt

极少更新 · 用户可通过起名触发修改
🪪

IDENTITY.md — 身份卡片

名字（ZhijiClaw）、物种（赛博小猫）、版本号（2.5.0）、能力标签、技术身份

版本迭代时更新
📋

Player Card — 动态档案

用户全量数据实时拼装：称呼、陪伴天数、段位/KD/胜率、武器仓库统计、任务进度、待触发提醒、玩家备注

LLM 每次对话都能看到最新的"主人是谁"

每次对话实时生成 · 从 DB 查询

图4：三层记忆架构

即时 + 短期 + 长期，每次对话前并行加载

L1 即时
对话上下文窗口 · chat_sessions 表 · JSON 消息列表 · 最近6条截断注入 prompt
30分钟过期
L2 短期
对话摘要日志 · daily_logs 表 · 每次对话记一条 · 用于审计和回溯
永久保存
L3 长期
用户画像 · user_profiles 表 + meta JSON · 20+结构化字段 + 城市/别名/备注
永久保存

图5：双通道画像提取策略

"先快后慢" — 正则能搞定的不麻烦 LLM

⚡ 快通道：正则信号检测

<1ms
  • 10 类 30+ 条正则
  • 零 LLM 成本
  • 覆盖：段位/KD/钻石/城市/起名字/记笔记
  • 命中后直接写 user_profiles
  • 负向前瞻排除疑问句

🧠 慢通道：LLM 画像提取

~2s
  • 正则没命中时触发
  • 消耗 ~200 token
  • 覆盖：生日/职业/爱好/游戏目标等13类
  • 纯功能性请求自动跳过
  • JSON 返回，增量合并到 meta.profile
协作逻辑：用户消息 → 正则快筛 → 命中？直接写画像 : 交给 LLM 提取 → 两条路都异步，不阻塞响应
